## 自动发送逻辑分析

### 核心设计：智能自动发送系统

**设计理念**：
- **手打输入**：等待用户停止输入后自动发送（延迟发送）
- **粘贴/语音**：立即发送，不等待延迟
- **中文输入法**：通过 composition 事件检测组合输入，避免误发

### 关键技术点

#### 1. 中文输入法检测
```javascript
// compositionstart - 开始组合输入（如拼音输入法）
messageInput.addEventListener('compositionstart', () => {
    this.isComposing = true;
    this.clearAutoSendTimer();
});

// compositionend - 结束组合输入
messageInput.addEventListener('compositionend', (e) => {
    this.isComposing = false;
    this.handleSmartAutoSend(e);
});
```

**作用**：防止在输入拼音等组合输入时触发自动发送

#### 2. 空格检测机制
```javascript
const lastChar = value[value.length - 1];
const isLastCharSpace = lastChar === ' ';

if (isLastCharSpace) {
    // 最后一个字符是空格，不自动发送，等待用户继续输入
    this.clearAutoSendTimer();
}
```

**作用**：用户输入空格表示还想继续输入，暂缓自动发送

#### 3. 自动发送延迟
```javascript
const autoSendDelay = parseFloat(settings.autoSendDelay || 2.5); // 默认 2.5 秒
const delayMs = autoSendDelay * 1000;

this.autoSendTimer = setTimeout(() => {
    // 延迟后自动发送
    this.sendMessage();
}, delayMs);
```

**用户设置**：0.5-3 秒可调

#### 4. 计时器管理
```javascript
clearAutoSendTimer: function() {
    if (this.autoSendTimer) {
        clearTimeout(this.autoSendTimer);
        this.autoSendTimer = null;
    }
}
```

**触发时机**：
- 每次输入都重置计时器
- 最后一个字符是空格时清除
- 用户手动发送时清除

### 工作流程

```
用户输入 → isComposing检查 → 最后字符检查 → 重置计时器 → 延迟等待 → 自动发送
                                              ↓
                                          输入/空格 → 清除计时器 → 重新开始
```

### 特殊处理

1. **空输入**：清除计时器，不发送
2. **组合输入中**：跳过自动发送逻辑
3. **最后字符为空格**：取消自动发送，等待继续输入
4. **手动发送/Enter**：清除计时器，立即发送
5. **粘贴操作**：立即发送（通过 input 事件检测）

### 优点

✅ **中文友好**：完美支持中文输入法组合
✅ **智能判断**：通过空格检测用户意图
✅ **可配置延迟**：0.5-3 秒自由调整
✅ **防误发**：多重检查机制
✅ **手动优先**：随时可手动发送打断
✅ **输入体验好**：不需要频繁点击发送按钮

### 配置项

- **autoSendDelay**：自动发送延迟（默认 2.5 秒）
- 设置界面：`设置 → 对话设置 → 自动发送延迟`
- 范围：0.5-3.0 秒

## 性能问题分析

**时间**: 2026-03-24 10:50

### 用户反馈
AI 发来的文本经过拆解后，分别以语音和情景文字方式发布，但节奏缓慢，用户体验较差。

### 问题定位

**消息格式**：
```
[温馨的卧室] "亲爱的，你今天过得怎么样？"
[她温柔地看着你] "我很想念你"
```

**当前播放流程**（每条消息）：
1. 显示场景 → 等待 800ms（固定）
2. 淡出场景
3. 播放语音（等待完成）
4. 等待 200ms 间隔

**问题**：
- 场景和语音串行处理，不能并行
- 800ms 固定延迟太长，短场景也等待 800ms
- 消息间隔固定 200ms，缺乏灵活性

### 改进方案

#### 方案 1：场景与语音并行（推荐，P0）
**改动**：场景显示后立即开始播放语音，400ms 后淡出场景
**效果**：每条消息节省 400ms 延迟
**实施难度**：低

#### 方案 2：动态场景显示时间（P1）
**改动**：根据场景文字长度动态计算显示时间
**效果**：短场景快速显示，长场景有足够时间
**实施难度**：低

#### 方案 3：优化消息间隔（P2）
**改动**：根据下一条是否有场景动态调整间隔时间
**效果**：减少不必要的等待
**实施难度**：低

### 效果预估

**当前**（3 条消息）：
- 总耗时：约 9.3 秒

**改进后**（3 条消息，实施 P0+P1+P2）：
- 总耗时：约 5-6 秒
- 节省：约 40-50% 的时间

### 建议行动
立即实施 P0（场景与语音并行），这是最简单且效果最明显的改进。

## 串行模式 TTS 播放错误修复

**时间**: 2026-03-24 11:20

### 问题

当 `useParallelSceneAndSpeech = false`（串行模式）时，TTS 播放出现 `interrupted` 错误。

### 根本原因

`TTS.speak()` 函数在调用 `synth.speak()` 后，依赖异步的 `onstart` 回调来设置 `this.isPlaying = true`。但在 `app.js` 中，调用 `TTS.speak()` 后立即进入等待循环检查 `TTS.isPlaying`，此时 `onstart` 回调可能还未执行，导致 `isPlaying` 为 `false`，等待循环立即退出。

**时序问题**：
```
TTS.speak() 调用
  ↓
this.synth.speak(utterance) 返回（同步）
  ↓
app.js 立即检查 TTS.isPlaying（此时为 false）
  ↓
等待循环立即退出
  ↓
下一条消息调用 TTS.speak()
  ↓
this.stop() 被调用 → 中断前一条语音
```

### 解决方案

在 `TTS.speak()` 函数中，在调用 `synth.speak()` **之前**就设置 `this.isPlaying = true`，确保调用者能立即检测到。

**修改文件**: `js/tts.js`

**修改内容**：
```javascript
// 修改前
this.currentUtterance = utterance;
this.synth.speak(utterance);

// 修改后
this.currentUtterance = utterance;
// 在 speak 之前就设置 isPlaying，确保调用者能立即检测到
this.isPlaying = true;
this.synth.speak(utterance);
```

### 效果

✅ 串行模式现在能正确等待每条语音播放完成
✅ 不再出现 `interrupted` 错误
✅ 两种模式（串行/并行）都能正常工作
✅ 不影响并行模式的功能

### 测试验证

修复后，可以通过控制台测试：
```javascript
// 测试串行模式
App.useParallelSceneAndSpeech = false;

// 发送一条长消息，观察是否正常播放
// 不应出现 interrupted 错误
```
